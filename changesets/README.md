# changesets/ — 变更记录（需求/PRD 落盘）

将产品需求、用户故事及其变更历史以代码形式管理，使 AI Agent 能理解"为什么要做这个改动"。

## 定位

- 记录**业务需求**的原始描述和演进过程
- 提供**变更意图**，让 AI Agent 在编码时理解目标而非只看结果
- 形成**可追溯链**：需求 → 设计 → 代码 → 测试

## 内容范围

- **需求文档** —— PRD、用户故事、验收标准
- **需求变更** —— 变更原因、影响范围、变更前后对比
- **设计方案** —— 功能设计、交互方案（非技术架构，技术架构放 `tech-knowledge/`）

## 目录组织建议

```
changesets/
├── YYYY-MM/                    # 按月归档
│   ├── 需求名称/
│   │   ├── prd.md             # 需求文档
│   │   ├── design.md          # 设计方案
│   │   └── changelog.md       # 变更记录
│   └── ...
└── README.md
```

## 文档模板

```markdown
# [需求名称]

> 版本：v1.0 | 日期：YYYY-MM-DD | 负责人：xxx

## 背景

为什么要做这个需求，解决什么问题。

## 目标

可量化的目标或验收标准。

## 功能描述

具体要做什么（用户视角）。

## 影响范围

涉及哪些模块、接口、数据。

## 变更历史

| 日期 | 变更内容 | 原因 |
|------|----------|------|
| ... | ... | ... |
```

## 注意事项

- 需求变更时**追加记录**，不要覆盖原始描述——历史对 AI 理解演进非常重要
- 验收标准要足够具体，AI Agent 可据此生成测试用例
- 与 git commit 的区别：commit 记录"改了什么"，这里记录"为什么要改"
