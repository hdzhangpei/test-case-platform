## 四、测试用例设计

### 4.1 整体结构

```
tests/
├── integration/
│   ├── requirements/                    # 按需求管理
│   │   ├── digital-user-management/     # 岗位虾一期
│   │   │   ├── scenarios/               # 场景目录
│   │   │   │   ├── position-shrimp-add/ # 场景1：添加岗位虾
│   │   │   │   ├── position-shrimp-edit/# 场景2：编辑岗位虾
│   │   │   │   └── ...
│   │   │   └── config.json              # 需求配置
│   │   └── ...
│   │
│   ├── regression/                      # 项目级测试用例汇总（系统回归）
│   │   ├── scenarios/                   # 归档后的场景用例
│   │   │   ├── position-shrimp-add/     # 已归档场景
│   │   │   └── ...
│   │   └── regression-config.json       # 回归配置
│   │
│   └── test-config.json                 # 全局配置
```

### 4.2 场景定义

**场景 = 用户登录后针对核心接口变更，按正常页面操作模拟所有关键请求**

每个场景包含：
- **目标接口正常主流程**：验证核心功能正确性
- **异常流验证**：验证边界条件和错误处理

### 4.3 场景示例：添加岗位虾

```
场景：position-shrimp-add（添加岗位虾）
描述：用户登录后，从进入页面到成功添加岗位虾的完整流程

步骤编排：
  1. 加载岗位类型枚举（页面初始化）
  2. 加载岗位列表（页面初始化）
  3. 查询岗位虾列表（页面加载）
  4. 添加岗位虾（核心接口-正常流）
  5. 验证添加结果（断言）
  6. 添加虾-agentId重复（核心接口-异常流）
  7. 添加虾-miId重复（核心接口-异常流）
  8. 添加虾-跨领域校验（核心接口-异常流）
  9. 添加虾-岗位重复（核心接口-异常流）
```

### 4.4 占位符语法

```yaml
# 引用步骤4的输入参数
value: "${step:4:input.userInfo.agentId}"

# 引用步骤4的返回数据
value: "${step:4:data.agentId}"
```

### 4.5 用例归档机制

```
需求上线
    ↓
需求用例归档到 regression/scenarios/
    ↓
后续需求修改
    ↓
新增用例 + 存量用例不受影响
    ↓
系统级回归测试
```

**原则**：
- 每次上线后，需求用例归档到 `regression/scenarios/`
- 归档后的用例用于系统级回归测试
- 后续修改不能影响存量用例
- 新增用例可以引用存量用例的返回值

## 六、原则

1. **一个项目完美闭环就是胜利** - 不用覆盖所有场景
2. **该人工的地方果断人工** - Review环节必须人工
3. **AI不可能100%准确** - 流程必须可调整
4. **在某个环节深度打磨** - 用例Review是重点
5. **存量用例不能受影响** - 归档后的用例必须保持稳定
